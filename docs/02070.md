## Azure 架构简介

如果在软件中模拟计算机硬件，则称为虚拟化，Microsoft azure 基于这种技术，其中通过映射软件指令来模拟硬件指令，这样，虚拟化的硬件可以通过使用软件来执行类似“真实”硬件的功能，Microsoft Azure 的架构运行在庞大的服务器集合和网络硬件上，这些硬件能够托管复杂的应用程序集合，并可以控制这些服务器上的软件操作和配置。

### 蔚蓝的建筑

azure 架构的不同架构风格解释如下:

#### 1.多层架构

下图显示了微软 azure 中的 N 层架构。

![N-tier architecture](../Images/7694e5661af5dad342d0894b9f346dd4.png)

<noscript><img class="alignnone size-full wp-image-321971" src="../Images/7694e5661af5dad342d0894b9f346dd4.png" alt="N-tier architecture" width="631" height="309" srcset="https://cdn.educba.com/academy/wp-content/uploads/2020/03/Azure-Architecture-1.jpg 631w, https://cdn.educba.com/academy/wp-content/uploads/2020/03/Azure-Architecture-1-300x147.jpg 300w" sizes="(max-width: 631px) 100vw, 631px" data-original-src="https://cdn.educba.com/academy/wp-content/uploads/2020/03/Azure-Architecture-1.jpg"/></noscript>

在企业应用中使用 n 层架构，因为它是一种传统的架构。应用程序分为几层，每层执行逻辑功能，如业务中使用的逻辑等。这样，依赖性就被管理了。但是有一个负债，那就是水平层。如果我们想在应用程序的任何一个部分进行更改，我们也必须在应用程序的其他部分进行一些更改。因此，存在频繁更新的挑战，并且不容易添加新功能。如果我们想迁移基于分层架构的现有应用程序，那么 N 层架构是一个很好的选择。

#### 2.Web 队列工作程序

下图显示了 Microsoft azure 中的 web 队列工作器架构。

![Web queue worker](../Images/9522d547df01dffe2d50515d1ffdefaa.png)

<noscript><img class="alignnone size-full wp-image-321972" src="../Images/9522d547df01dffe2d50515d1ffdefaa.png" alt="Web queue worker" width="615" height="275" srcset="https://cdn.educba.com/academy/wp-content/uploads/2020/03/Azure-Architecture-2.jpg 615w, https://cdn.educba.com/academy/wp-content/uploads/2020/03/Azure-Architecture-2-300x134.jpg 300w" sizes="(max-width: 615px) 100vw, 615px" data-original-src="https://cdn.educba.com/academy/wp-content/uploads/2020/03/Azure-Architecture-2.jpg"/></noscript>

使用这种架构风格的应用程序由一个处理 HTTP 请求的前端和一个执行 CPU 密集型操作的后端组成。异步消息队列用于前端和后端之间的通信。这种风格的架构适合于基于资源的简单任务。使用托管服务简化了部署和操作。但是如果领域很复杂，那么管理依赖关系就很困难。对于复杂的域，前端和后端的维护和更新变得困难。

#### 3.微服务架构

下图显示了微软 azure 中的微服务架构。

![Microservices Architecture](../Images/c9083a6a0232e24b423d860fe90749f7.png)

<noscript><img class="alignnone size-full wp-image-321973" src="../Images/c9083a6a0232e24b423d860fe90749f7.png" alt="Microservices Architecture" width="558" height="327" srcset="https://cdn.educba.com/academy/wp-content/uploads/2020/03/Azure-Architecture-3.jpg 558w, https://cdn.educba.com/academy/wp-content/uploads/2020/03/Azure-Architecture-3-300x176.jpg 300w" sizes="(max-width: 558px) 100vw, 558px" data-original-src="https://cdn.educba.com/academy/wp-content/uploads/2020/03/Azure-Architecture-3.jpg"/></noscript>

许多小而独立的服务构成了微服务架构。单个业务能力由每个服务实现。服务通过 API 的契约进行通信，它们是松散耦合的。专注的开发团队可以构建服务。团队之间的协调对于构建单独的服务是必要的，并且鼓励频繁的更新。与 N-tier 和 web queue worker 等其他架构相比，微服务架构的构建非常复杂。良好的 devops 开发和文化是构建微服务架构的必要条件。但是如果微服务架构以正确的方式构建，这种架构风格会带来更高的速度和更快的创新。

#### 4.命令和查询责任分离体系结构

下图显示了微软 azure 中的 CQRS 架构。

![Command and Query Responsibility Segregation Architecture](../Images/4b83d9dbe5119391f89b93fd102061cc.png)

<noscript><img class="alignnone size-full wp-image-321974" src="../Images/4b83d9dbe5119391f89b93fd102061cc.png" alt="Command and Query Responsibility Segregation Architecture" width="376" height="294" srcset="https://cdn.educba.com/academy/wp-content/uploads/2020/03/Azure-Architecture-4.jpg 376w, https://cdn.educba.com/academy/wp-content/uploads/2020/03/Azure-Architecture-4-300x235.jpg 300w" sizes="(max-width: 376px) 100vw, 376px" data-original-src="https://cdn.educba.com/academy/wp-content/uploads/2020/03/Azure-Architecture-4.jpg"/></noscript>

读取和写入操作通过命令和查询责任分离架构被分离到两个不同的模型中。因此，系统中写入数据或更新数据的部分被系统中读取数据的部分分开。此外，写操作是在单独的数据库上执行的，读操作是针对实例化视图执行的。这有助于独立执行读写操作，并且可以针对查询优化物化视图。命令和查询责任分离体系结构是一些大型体系结构的良好子集，因为当它应用于应用程序时，整个体系结构变得复杂。当任意数量的用户访问相同的数据时，可以为协作域考虑命令和查询责任分离架构。

#### 5.事件驱动架构

下图显示了微软 azure 中的事件驱动架构。

![Event Driven Architecture](../Images/7a3e5f184373e65eefddc83836527376.png)

<noscript><img class="alignnone size-full wp-image-321975" src="../Images/7a3e5f184373e65eefddc83836527376.png" alt="Event Driven Architecture" width="633" height="230" srcset="https://cdn.educba.com/academy/wp-content/uploads/2020/03/Azure-Architecture-5.jpg 633w, https://cdn.educba.com/academy/wp-content/uploads/2020/03/Azure-Architecture-5-300x109.jpg 300w" sizes="(max-width: 633px) 100vw, 633px" data-original-src="https://cdn.educba.com/academy/wp-content/uploads/2020/03/Azure-Architecture-5.jpg"/></noscript>

事件驱动架构遵循发布-订阅模型，也称为发布-订阅模型。在这个模型中，事件由生产者发布，由消费者订阅。生产者和消费者相互独立，消费者也相互独立。例如，考虑处理大量数据且延迟非常低的物联网解决方案应用；然后，事件驱动架构非常有用。当存在相同的事件数据，并且不同的子系统对该相同的事件数据执行不同种类的处理时，这种架构也是有用的。

下图显示了微软 azure 中的大数据、大计算架构。

![Big Data, Big Compute Architecture](../Images/ad7a7203a3320073a7227d7e72a4daec.png)

<noscript><img class="alignnone size-full wp-image-321977" src="../Images/ad7a7203a3320073a7227d7e72a4daec.png" alt="Big Data, Big Compute Architecture" width="557" height="237" srcset="https://cdn.educba.com/academy/wp-content/uploads/2020/03/Azure-Architecture-6.jpg 557w, https://cdn.educba.com/academy/wp-content/uploads/2020/03/Azure-Architecture-6-300x128.jpg 300w" sizes="(max-width: 557px) 100vw, 557px" data-original-src="https://cdn.educba.com/academy/wp-content/uploads/2020/03/Azure-Architecture-6.jpg"/></noscript>

一些配置文件仅适合特定类型的体系结构，其中一种体系结构是大数据、大计算体系结构，这是适合特定类型配置文件的特殊体系结构之一。一个非常庞大的数据集被大数据划分为多个块，用于并行处理数据以及报告和分析。大计算也称为高性能计算(HPC)，用于在数千个或更多内核上并行执行计算。模拟、三维渲染、建模等。，是大计算的领域。
