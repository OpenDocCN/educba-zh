## Teradata 和 Oracle 的区别

Teradata vs Oracle 是两个非常流行的 RDBMS 系统。对于 Oracle，RDBMS 变体是一个对象关系数据库管理系统(ORDBMS)。RDBMS 就像一个关系模型，它使用我们所说的索引、主键和外键来维护表之间的关系。因此，与旧的 DBMS 系统相比，获取和存储数据的速度更快。Oracle 面向对象的数据库模型利用数据库模式和查询语言支持的对象和类。Oracle 在 70 年代第一个将 RDBMS 商业化，当时 Teradata 只是在为第一个数据仓库打基础。后来，Teradata 的能力使其最适合大数据、[商业智能工具](https://www.educba.com/business-intelligence-tool/)以及物联网。

### Teradata 和 Oracle 的主要区别

Teradata vs Oracle 都是市场上的热门选择；让我们讨论一些主要的区别:

*   Oracle 是为事务处理量身定制的系统，因为它具有很大的架构灵活性，而 Teradata 的 OLAP 非常强大，可以对数据进行切片和切块以供分析。
*   Oracle 主要用作在线后端应用程序。它管理事务中的插入、更新和删除，而 Teradata 是数据仓库应用程序，用于维护大数据以进行分析。Teradata 中没有实时事务这种东西。
*   Teradata 可以被视为硬件和软件之间的良好结合，从而产生了高端企业数据库设备。但是甲骨文在 2008 年推出了它的 OLAP 数据库云服务器，相比之下，这是相当晚的。
*   Teradata 是基于无共享架构，另一方面 Oracle 有共享一切架构。无共享体系结构是指多处理器数据库管理系统，在该系统中，处理器之间不共享内存和硬盘存储，仅共享网络带宽用于数据传输。
*   对于 Teradata，数据存储在服务器上，并在一组服务器上进行分区，其中每台服务器只负责自己的数据。就 Oracle 而言，该架构意味着任何机器都可以访问任何数据。这一根本区别使得 Teradata 非常适合数据仓库，而 Oracle 适合 OTLP。
*   Teradata 作为一个设备非常适合数据聚合。因为它是一种电器，所以获得更多存储或更大功率的唯一方法是购买更多电器。Teradata 的实现要求投资回报不能强迫。
*   Teradata 缺少一个好的、复杂的数据抽象层。这就让人把它当成了只读数据源。如果有一个简单的数据仓库应用程序，只需要生成报告之类的东西，这仍然很好。此外，Teradata 没有最智能的优化器来处理它。它可以完美地处理一些模型。有时，基于查询的性质，复杂的星型模式可能会导致混淆并花费大量时间。规模递归模型超出预期。
*   相反，甲骨文是任何东西，如果有人有能力创造，那么就能做到。对于为几千兆字节的企业信息工厂设置做数据基础的应用服务器来说，这是一个非常便宜的选择。它可以在从 SAN 到操作系统或数据库甚至抽象层的每个级别进行定制。通常，任何对 RDBMS 有意义的模型都可以与任何复杂的优化器一起工作。一个缺点是 Oracle 对于不知道如何正确使用它的人来说很难，这是因为有太多的定制选项用于太多不同的用例，这需要很长的学习曲线。如果事情不按照神谕的方式去做，就不会顺利。Oracle 也存在可扩展性问题，如果数据量用完了可用的存储空间，就需要购买更多的硬件，这会导致巨大的成本。

### 结论

最后，我们可以说 Teradata 和 Oracle 系统都存在可伸缩性问题。Teradata 的一个问题是，对于数据频繁更新的大规模系统，保持更新变得非常昂贵。因此，Teradata 和 Oracle 系统都需要某种策略来解决可伸缩性问题。除此之外，这些系统都为客户提供了巨大的好处。
